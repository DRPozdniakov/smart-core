Section 2

Document Ingestion Pipeline

How markdown files transform into graph nodes with embeddings and entity relationships.

1

Read Markdown File

Load .md file from docs_ver2/ folder with YAML front matter.

2

Parse YAML Metadata

Extract doc_id, title, owner, status, domain, phase, tags, dependencies.

3

Chunk by Headers

Split content at markdown headers (##, ###) for precise retrieval.

4

Generate Embeddings

Create 1536-dim vectors using OpenAI ada-002 for semantic search.

5

Extract Entities

Identify canonical entities (Round-Seed, Prod-FersHumanoid) and tags.

6

Store in Neo4j

Create Document, Chunk, Entity nodes with CONTAINS, HAS_ENTITY relationships.

⚡
Efficient Reloading:

Smart Core computes SHA-256 hashes of file contents. Only changed files are re-ingested, making frequent reloads fast (~5-10 seconds for typical edits).

Section 3

Hybrid Search Flow

Query engine combines vector similarity (semantic) with graph traversal (entity-based) for best-of-both retrieval.

1

Vector Search

Embed user query, compute cosine similarity against chunk embeddings, return top K matches.

2

Graph Traversal

Match entities/tags in query, follow HAS_ENTITY and ALSO_IN relationships to find relevant chunks.

3

Merge & Rank

Combine results from both paths, deduplicate by chunk ID, rank by combined score.

4

Return to Claude

Send chunks with source attribution (doc_id, section) and entity metadata for reasoning.

🚀
Performance Advantage:

Hybrid search delivers 62.5% fewer tool calls (6 vs 16) compared to sequential file reading. Vector search finds semantically similar content, while graph traversal ensures entity consistency across documents.

Section 4

Change Propagation Workflow

When an entity value changes (e.g., funding amount, robot price), Smart Core automatically detects affected documents and requests user approval before propagating updates.

1

Edit Document

User changes an entity value (e.g., funding amount in Financial Model).

2

Detect Drift

Smart Core compares file hashes and detects which documents changed.

3

Find Affected

Query graph for all documents containing the changed entity via ALSO_IN relationships.

4

Create Merge Request

Generate MergeRequest node with source, entity, old/new values, target documents.

5

User Approval

Claude asks user via AskUserQuestion: "Entity: old→new. Approve?" User decides.

6

Propagate Changes

If approved, approve_merge() triggers cross-document updates.

7

Update All Files

Edit all target documents with new entity value in one atomic operation.

8

Commit Changes

Record git-style commit with author, message, and change details in Neo4j.

9

Reload Graph

Re-ingest changed documents to sync graph state with files.

🔒
Human-in-the-Loop Safety:

Smart Core NEVER auto-approves merge requests. Every cross-document change requires explicit user approval. This prevents accidental propagation of incorrect values across critical business documents.

Section 5

Multi-Instance Collaboration

How multiple team members (CTO, CFO, CEO) work concurrently on different documents while maintaining shared context through Neo4j.

1

Independent Sessions

Each team member has their own Claude Code session. No file locking or coordination needed.

2

Shared Graph

All sessions connect to the same Neo4j database. Commits from one user are visible to all.

3

Change Awareness

get_commit_history() shows who changed what, when. Each session can query recent activity.

4

Conflict Detection

If two users edit the same entity simultaneously, Smart Core flags the conflict for resolution.

Graph schema showing node types and relationships that power Smart Core's knowledge graph.

title STRING

owner STRING
status ENUM
file_hash STRING
Chunk
chunk_id PK STRING
doc_id STRING
text STRING
embedding FLOAT[1536]
section STRING
Entity
entity_id PK STRING
type STRING
name STRING
value STRING
last_updated DATE
Commit
commit_id PK STRING
author STRING
date TIMESTAMP
message STRING
📊
Graph-First Design:

Unlike traditional document stores, Smart Core models relationships as first-class citizens. The ALSO_IN relationship enables instant cross-document entity tracking, powering change propagation and consistency validation.
